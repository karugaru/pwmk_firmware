# ビルドテスト手順

## 動作確認済みのテスト手順

ここでは、実際にビルドテストを行った手順を紹介します。

### VS Code

 VS Code タスクの`Compile Project`でビルド

### CLI (local)

WSL2 上で`uv run tools/pwmk.py build`でビルド

### CLI (docker)

WSL2 上で `uv run tools/test_build.py ubuntu_26_04`でビルドテストを実行

## Docker を使ったビルドテストについて

Docker を使って CLI ビルドを検証する手順を説明します。
`test_build.py` は、各ディストリビューションの公式コンテナを起動し、コンテナ内で `uv run tools/pwmk.py build` を実行してビルドを検証します。

### 1. 前提環境

この手順は次の前提で動作します。

- Linux 環境に入っていること
- Dockerと`uv` が使えること

ビルドテストでは APT パッケージ取得を高速化するために、`apt-cacher-ng` のコンテナを自動的に起動して利用します。

### 2. ビルドテストの実行

リポジトリルートで次のコマンドを実行します。

```bash
uv run tools/test_build.py
```

特定のディストリビューションだけをテストしたい場合は、ディストリビューション名を引数として指定します。

```bash
uv run tools/test_build.py ubuntu_26_04
uv run tools/test_build.py debian_12_15 fedora_43
```

実行を開始すると確認なしでビルドが進行するのでご注意ください。

このスクリプトは次のディストリビューションを順番に検証します。

- Ubuntu
- Debian
- AlmaLinux
- Fedora
- Arch Linux

このスクリプトは各ディストリビューションごとに次の処理を行います。

- `pwmk-build-test-network` ネットワークを作成し、必要であれば `pwmk-apt-cacher-ng` コンテナを起動する
- 対象ディストリビューションの公式コンテナを起動する
- ソースツリーを `/workspace` にマウントする
- `pwmk-pico-sdk-cache-<distribution>` ボリュームを `/root/.pwmk` にマウントし、対象ディストリビューションごとに `pico-sdk` / `picotool` の取得結果とビルド成果物を再利用する
- Ubuntu と Debian では、コンテナ内の APT を `apt-cacher-ng` 経由に設定する
- コンテナ内で `uv` を導入する
- コンテナ内で `uv sync --frozen` を実行する
- コンテナ内で `uv run tools/pwmk.py build ...` を実行する

各ディストリビューションの結果は個別に表示され、途中で失敗しても残りのディストリビューションの検証は継続されます。最後に成功・失敗のサマリを表示します。

テスト実行中の詳細ログは `tools/log` 配下に保存され、各ディストリビューションごとのログ、準備処理ログ、サマリログを確認できます。

### 3. 補足

APT キャッシュを消したい場合は、必要に応じて Docker 側で `pwmk-apt-cacher-ng` コンテナと `pwmk-apt-cacher-ng-cache` ボリュームを削除してください。

`pico-sdk` / `picotool` の取得キャッシュを消したい場合は、対象ディストリビューション用の `pwmk-pico-sdk-cache-<distribution>` ボリュームを削除してください。これを残しておくと、同じ `SDK_TAG` での再実行時に clone やビルドを省略できます。

### 4. 期待する結果

各ディストリビューションでビルドが成功すると、そのコンテナのログ末尾で `pwmk.elf` のリンク完了とメモリ使用量が表示されます。
